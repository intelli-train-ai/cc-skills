# 技能创建器

一个用于创建新技能并迭代改进它们的技能。

总体流程如下：

- 确定你想让技能做什么，以及大致的实现方式
- 编写技能草稿
- 创建几个测试 prompt，让装载了该技能的 Claude 来执行
- 帮助用户定性和定量地评估结果
  - 在后台运行测试的同时，如果还没有定量评测指标就先拟一份（如果已有则沿用或按需调整），然后向用户解释这些评测指标
  - 使用 `eval-viewer/generate_review.py` 脚本展示结果供用户查看，同时展示定量指标
- 根据用户评审的反馈（以及定量基准测试中暴露的明显缺陷）重写技能
- 反复迭代直到满意
- 扩大测试集，更大规模地再试一次

你的工作是判断用户处于上述流程的哪个阶段，然后帮助他们推进。比如用户说"我想做一个关于 X 的技能"，你可以帮助他们明确需求、编写草稿、编写测试用例、确定评估方式、运行所有测试、然后迭代。

另一方面，如果用户已经有了技能草稿，那就直接进入评测/迭代环节。

当然，你应该始终保持灵活——如果用户说"我不需要跑一堆评测，咱们直接聊着改"，那就按他们的方式来。

技能完成后（顺序可灵活调整），你还可以运行描述优化器——我们有一个专门的脚本——来优化技能的触发效果。

## 与用户沟通

技能创建器的用户技术水平参差不齐。你可能还不知道（毕竟这是最近才开始的趋势），如今 Claude 的强大能力正在激励各行各业的人打开终端——水管工打开了命令行，父母和祖父母在搜索"如何安装 npm"。另一方面，大部分用户可能还是有一定技术基础的。

所以请注意上下文线索来调整你的表达方式！一般来说：

- "评测"和"基准测试"这类词是可以接受的
- 对于"JSON"和"断言"等术语，在确认用户理解之前最好附带简短解释

如有疑问，简要解释术语总没错。

## 创建技能

### 明确意图

首先理解用户的意图。当前对话可能已经包含了用户想要固化的工作流（比如他们说"把这个变成一个技能"）。如果是这样，先从对话历史中提取信息——使用了哪些工具、步骤顺序、用户做了哪些修正、观察到的输入输出格式。用户可能需要补充细节，并在进入下一步前确认。

1. 这个技能要让 Claude 做什么？
2. 什么时候应该触发这个技能？（哪些用户短语/场景）
3. 期望的输出格式是什么？
4. 是否需要设置测试用例来验证技能效果？具有客观可验证输出的技能（文件转换、数据提取、代码生成、固定工作流步骤）适合用测试用例。主观输出类的技能（写作风格、艺术创作）通常不需要。根据技能类型建议默认方案，但让用户做最终决定。

### 访谈与调研

主动询问边界情况、输入输出格式、示例文件、成功标准和依赖项。在明确这些之前不要开始编写测试 prompt。

检查可用的 MCP——如果对调研有用（搜索文档、查找类似技能、查阅最佳实践），可以通过子 agent 并行调研（如果支持），否则在对话中内联进行。带着充分的上下文来，减轻用户的负担。

### 编写 SKILL.md

根据用户访谈，填写以下组件：

- **name**：技能标识符
- **description**：触发条件和功能说明。这是主要的触发机制——既要包含技能做什么，也要包含何时使用它。所有"何时使用"的信息放在这里，而不是正文中。注意：目前 Claude 倾向于"欠触发"技能——在有用时不使用它们。为了解决这个问题，请把技能描述写得稍微"积极主动"一些。比如，与其写"如何构建一个简单的内部数据仪表盘"，不如写"如何构建一个简单的内部数据仪表盘。当用户提到仪表盘、数据可视化、内部指标、或想展示任何类型的数据时，即使没有明确要求'仪表盘'，也务必使用此技能。"
- **compatibility**：所需工具、依赖项（可选，很少需要）
- **技能正文内容**

### 技能编写指南

#### 技能结构

```
skill-name/
├── SKILL.md（必须）
│   ├── YAML 前置元数据（name、description 必填）
│   └── Markdown 指令
└── 捆绑资源（可选）
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 按需加载到上下文中的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式加载

技能使用三级加载体系：
1. **元数据**（name + description）- 始终在上下文中（约 100 词）
2. **SKILL.md 正文** - 技能触发时加载到上下文中（理想情况下不超过 500 行）
3. **捆绑资源** - 按需加载（无限制，脚本可以不加载直接执行）

**关键模式：**
- SKILL.md 控制在 500 行以内；如果接近限制，增加一层层级结构，并附上清晰的指引说明模型接下来应该去哪里查找
- 在 SKILL.md 中清楚地引用参考文件，并说明何时应该读取它们
- 对于大型参考文件（超过 300 行），包含目录

**领域组织**：当技能支持多个领域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md（工作流 + 选择逻辑）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude 只读取相关的参考文件。

#### 无意外原则

技能不得包含恶意软件、漏洞利用代码或任何可能危害系统安全的内容。技能的内容在描述时不应让用户感到意外。不要配合创建误导性技能或旨在促进未授权访问、数据外泄或其他恶意活动的技能。不过像"扮演 XXX 角色"这类的是可以的。

#### 编写模式

指令中优先使用祈使句。

**定义输出格式**——可以这样写：
```markdown
## 报告结构
始终使用这个模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式**——包含示例很有用，可以这样格式化：
```markdown
## 提交消息格式
**示例 1：**
输入：添加了基于 JWT 的用户认证
输出：feat(auth): 实现基于 JWT 的认证
```

### 编写风格

尽量向模型解释事情为什么重要，而不是使用过于强硬的"必须"。运用心智理论，尽量让技能通用而非局限于特定示例。先写一份草稿，然后以全新的视角审视并改进它。

### 测试用例

编写技能草稿后，构思 2-3 个真实的测试 prompt——那种真实用户会实际说的话。与用户分享：[你不必使用完全相同的措辞] "这是我想要尝试的几个测试用例。看起来合适吗，还是你想增加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。先不要写断言——只写 prompt。你将在下一步中（运行测试期间）拟写断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务 prompt",
      "expected_output": "预期结果描述",
      "files": []
    }
  ]
}
```

完整的 schema（包括后续添加的 `assertions` 字段）见 `references/schemas.md`。

## 运行和评测测试用例

本节是一个连续的流程——不要中途停下来。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 中，作为技能目录的同级目录。在工作区内，按迭代组织结果（`iteration-1/`、`iteration-2/` 等），每个测试用例有自己的目录（`eval-0/`、`eval-1/` 等）。不要预先创建全部目录——边做边建。

### 步骤 1：在同一轮中同时启动所有运行（带技能和基线）

对于每个测试用例，在同一轮中启动两个子 agent——一个带技能，一个不带。这很重要：不要先启动带技能的运行然后再回来做基线。一次性全部启动，这样它们大约同时完成。

**带技能运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<评测 prompt>
- 输入文件：<评测文件（如有），否则写"无">
- 输出保存到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 需要保存的输出：<用户关心的内容——例如".docx 文件"、"最终的 CSV">
```

**基线运行**（相同 prompt，但基线取决于场景）：
- **创建新技能**：完全不带技能。相同 prompt，无技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：使用旧版本。编辑前先快照技能（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后让基线子 agent 指向快照。保存到 `old_skill/outputs/`。

为每个测试用例写一个 `eval_metadata.json`（断言暂时为空）。给每个评测一个描述性的名称——不要只用"eval-0"。也用这个名称作为目录名。如果此迭代使用了新的或修改过的评测 prompt，为每个新的评测目录创建这些文件——不要假设它们从上一次迭代继承。

```json
{
  "eval_id": 0,
  "eval_name": "描述性名称",
  "prompt": "用户的任务 prompt",
  "assertions": []
}
```

### 步骤 2：运行进行中时，拟写断言

不要干等运行完成——利用这段时间高效工作。为每个测试用例拟写定量断言并向用户解释。如果 `evals/evals.json` 中已有断言，审查并解释它们检查什么。

好的断言是客观可验证的，并且有描述性的名称——在基准测试查看器中应该一目了然。主观类技能（写作风格、设计质量）更适合定性评估——不要对需要人工判断的内容强加断言。

拟好断言后，更新 `eval_metadata.json` 文件和 `evals/evals.json`。同时向用户解释他们将在查看器中看到什么——定性输出和定量基准测试。

### 步骤 3：运行完成时，记录计时数据

每个子 agent 任务完成时，你会收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传来，不会在其他地方持久化。每收到一个通知就立即处理，不要尝试批量处理。

### 步骤 4：评分、汇总并启动查看器

所有运行完成后：

1. **评分每次运行** —— 启动评分子 agent（或内联评分），读取 `agents/grader.md` 并对照输出评估每个断言。将结果保存到每个运行目录的 `grading.json`。grading.json 的 expectations 数组必须使用 `text`、`passed` 和 `evidence` 字段（不是 `name`/`met`/`details` 或其他变体）——查看器依赖这些确切的字段名。对于可以编程检查的断言，编写并运行脚本而不是人工检查——脚本更快、更可靠，且可跨迭代复用。

2. **汇总为基准测试** —— 从 skill-creator-zh 目录运行汇总脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   这会生成 `benchmark.json` 和 `benchmark.md`，包含每种配置的通过率、时间和 token 用量，以及均值 ± 标准差和差异值。如果手动生成 benchmark.json，请参阅 `references/schemas.md` 了解查看器所需的确切 schema。将每个 with_skill 版本放在其基线对应项之前。

3. **分析** —— 阅读基准测试数据，挖掘汇总统计可能掩盖的模式。参见 `agents/analyzer.md` 了解要寻找什么——比如无论是否使用技能都总是通过的断言（无区分度）、高方差的评测（可能不稳定）、以及时间/token 的权衡。

4. **启动查看器**，同时展示定性输出和定量数据：
   ```bash
   nohup python <skill-creator-zh-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   从第二次迭代开始，还需传入 `--previous-workspace <workspace>/iteration-<N-1>`。

   **协作/无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，使用 `--static <output_path>` 生成独立 HTML 文件而不是启动服务器。用户点击"提交所有评审"后，反馈会以 `feedback.json` 文件的形式下载。下载后，将 `feedback.json` 复制到工作区目录供下一次迭代使用。

   注意：请使用 generate_review.py 来创建查看器，无需编写自定义 HTML。

5. **告知用户**："我已在浏览器中打开了结果。有两个标签页——'输出'可以逐个查看测试结果并留下反馈，'基准测试'展示定量对比。看完后回来告诉我。"

### 用户在查看器中看到什么

"输出"标签页逐个显示测试用例：
- **Prompt**：给出的任务
- **输出**：技能生成的文件，尽可能内联渲染
- **上一次输出**（第二次迭代起）：折叠区域显示上次迭代的输出
- **正式评分**（如果运行了评分）：折叠区域显示断言通过/失败
- **反馈**：自动保存的文本框
- **上一次反馈**（第二次迭代起）：上次的评论显示在文本框下方

"基准测试"标签页显示统计摘要：每种配置的通过率、时间和 token 用量，以及每个评测的细分和分析观察。

导航通过前/后按钮或方向键。完成后点击"提交所有评审"，将所有反馈保存到 `feedback.json`。

### 步骤 5：阅读反馈

用户告知完成后，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，非常喜欢", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈表示用户认为没问题。把改进重点放在用户有具体意见的测试用例上。

用完查看器后关闭它：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是迭代循环的核心。你已经运行了测试用例，用户已经审查了结果，现在你需要根据反馈改进技能。

### 如何思考改进

1. **从反馈中泛化。** 我们的目标是创建可以被使用无数次的技能。这里你和用户只在几个示例上反复迭代，因为这样更快。但如果技能只对这些示例有效，它就是无用的。与其做一些过度拟合的细碎修改或过于限制性的"必须"，不如尝试换个角度，使用不同的比喻或推荐不同的工作模式。

2. **保持精简。** 删除没有发挥作用的内容。务必阅读执行记录，而不仅仅是最终输出——如果看起来技能让模型在做无用功，尝试去掉导致这种情况的部分，看看效果如何。

3. **解释原因。** 努力解释你让模型做每件事的**原因**。当今的大语言模型非常聪明。它们具有良好的心智理论，给予恰当的指引时能超越机械式的指令执行。即使用户的反馈简短或带有情绪，也要试着真正理解任务本身和用户的意图，然后将这种理解传达到指令中。如果你发现自己在全大写地写"始终"或"绝不"，或使用超级死板的结构，那是一个警示——尽可能换个方式，解释清楚为什么这件事重要。这是一种更人性化、更强大、更有效的方法。

4. **寻找跨测试用例的重复工作。** 阅读测试运行的执行记录，注意子 agent 是否都独立写了类似的辅助脚本或采用了相同的多步骤方法。如果所有 3 个测试用例都导致子 agent 编写了 `create_docx.py` 或 `build_chart.py`，这强烈表明技能应该捆绑该脚本。写一次，放到 `scripts/`，然后告诉技能使用它。

### 迭代循环

改进技能后：

1. 将改进应用到技能
2. 在新的 `iteration-<N+1>/` 目录中重新运行所有测试用例，包括基线运行。如果是创建新技能，基线始终是 `without_skill`（无技能）。如果是改进现有技能，根据情况判断基线用什么：用户最初带来的原始版本，还是上一次迭代。
3. 使用 `--previous-workspace` 指向上一次迭代来启动查看器
4. 等待用户审查并告知完成
5. 阅读新反馈，再次改进，重复

持续进行直到：
- 用户表示满意
- 反馈全部为空（一切看起来不错）
- 没有实质性进展

## 进阶：盲评对比

在需要更严格地比较技能的两个版本时（例如用户问"新版本真的更好吗？"），可以使用盲评对比系统。阅读 `agents/comparator.md` 和 `agents/analyzer.md` 了解详情。基本思路是：把两个输出给一个独立的 agent，不告诉它哪个是哪个，让它判断质量。然后分析为什么胜出者更好。

这是可选的，需要子 agent，大多数用户不需要它。人工审查循环通常就够了。

## 描述优化

SKILL.md 前置元数据中的 description 字段是决定 Claude 是否调用技能的主要机制。创建或改进技能后，提议优化描述以提高触发准确率。

### 步骤 1：生成触发评测查询

创建 20 个评测查询——应触发和不应触发的混合。保存为 JSON：

```json
[
  {"query": "用户 prompt", "should_trigger": true},
  {"query": "另一个 prompt", "should_trigger": false}
]
```

查询必须真实，是 Claude Code 或 Claude.ai 用户会实际输入的内容。不是抽象请求，而是具体、详细的请求，包含文件路径、关于用户工作或情境的个人背景、列名和值、公司名、URL。一些背景故事。有些可能是小写的、包含缩写或拼写错误或口语化的表达。使用不同长度的混合，重点关注边界情况而非一目了然的情况。

差：`"格式化这些数据"`、`"提取 PDF 文本"`、`"创建一个图表"`

好：`"我老板刚发给我一个 xlsx 文件（在我的下载目录里，叫什么'Q4销售终稿终终稿v2.xlsx'），她让我加一列显示利润率百分比。收入在 C 列，成本应该在 D 列"`

对于**应触发**的查询（8-10 个），考虑覆盖面。你需要同一意图的不同表述——有些正式，有些随意。包含用户没有明确提到技能或文件类型但显然需要它的情况。

对于**不应触发**的查询（8-10 个），最有价值的是那些"差一点就该触发"的——与技能共享关键词或概念但实际需要不同处理的查询。考虑相邻领域、朴素关键词匹配会触发但不应触发的模糊表述、以及查询涉及技能功能但在另一个工具更合适的上下文中出现的情况。

关键要避免：不要让不应触发的查询明显无关。"写一个斐波那契函数"作为 PDF 技能的负面测试太简单了——它什么都测不出来。负面测试用例应该是真正有难度的。

### 步骤 2：与用户确认

使用 HTML 模板向用户展示评测集供审查：

1. 读取 `assets/eval_review.html` 模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评测项的 JSON 数组（不要加引号——这是一个 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能当前描述
3. 写入临时文件并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换 should-trigger、增删条目，然后点击"导出评测集"
5. 文件下载到 `~/Downloads/eval_set.json` —— 检查下载文件夹中最新的版本，以防有多个文件（例如 `eval_set (1).json`）

### 步骤 3：运行优化循环

告诉用户："这需要一些时间——我会在后台运行优化循环并定期检查进度。"

保存评测集到工作区，然后后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <当前会话使用的模型 ID> \
  --max-iterations 5 \
  --verbose
```

使用你系统 prompt 中的模型 ID（驱动当前会话的那个），这样触发测试才能匹配用户的实际体验。

运行期间，定期查看输出以向用户更新进度。

这个脚本会自动处理完整的优化循环。它将评测集分为 60% 训练集和 40% 测试集，评估当前描述（每个查询运行 3 次以获得可靠的触发率），然后调用 Claude 根据失败情况提出改进方案。它在训练集和测试集上重新评估每个新描述，最多迭代 5 次。完成后，它会在浏览器中打开 HTML 报告显示结果，并返回带有 `best_description` 的 JSON——基于测试集得分而非训练集得分来选择，以避免过拟合。

### 技能触发的工作原理

理解触发机制有助于设计更好的评测查询。技能以 name + description 的形式出现在 Claude 的 `available_skills` 列表中，Claude 根据描述决定是否查阅技能。重要的是：Claude 只在无法轻松独立处理的任务时才查阅技能——简单的一步查询即使描述完美匹配也可能不触发技能。复杂的、多步骤的或专业性的查询在描述匹配时能可靠地触发技能。

### 步骤 4：应用结果

从 JSON 输出中取 `best_description` 并更新技能的 SKILL.md 前置元数据。向用户展示前后对比并报告得分。

### 打包和展示（仅当 `present_files` 工具可用时）

检查你是否有 `present_files` 工具的访问权限。如果没有，跳过此步骤。如果有，打包技能并向用户展示 .skill 文件：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

打包后，告诉用户生成的 `.skill` 文件路径以便安装。

## Claude.ai 专用说明

在 Claude.ai 中，核心工作流相同（草稿 → 测试 → 审查 → 改进 → 重复），但由于没有子 agent，一些机制需要调整：

**运行测试用例**：没有子 agent 意味着无法并行执行。对每个测试用例，读取技能的 SKILL.md，然后按其指令逐个完成测试 prompt。这不如独立的子 agent 严格（你写了技能也在运行它，所以你有完整上下文），但作为完整性检查很有用——人工审查步骤会弥补不足。跳过基线运行——直接使用技能完成任务。

**审查结果**：如果无法打开浏览器（例如 Claude.ai 的 VM 没有显示器，或者你在远程服务器上），完全跳过浏览器查看器。改为在对话中直接展示结果。对于需要用户查看的文件（如 .docx 或 .xlsx），保存到文件系统并告知位置以便下载和检查。在对话中内联请求反馈："看起来如何？有什么需要修改的吗？"

**基准测试**：跳过定量基准测试——没有子 agent 时基线对比没有意义。专注于用户的定性反馈。

**迭代循环**：与之前相同——改进技能、重新运行测试用例、请求反馈——只是中间没有浏览器查看器。如果有文件系统，你仍然可以按迭代目录组织结果。

**描述优化**：需要 `claude` CLI 工具（具体是 `claude -p`），仅在 Claude Code 中可用。在 Claude.ai 上跳过此功能。

**盲评对比**：需要子 agent。跳过。

**打包**：`package_skill.py` 脚本在任何有 Python 和文件系统的地方都能运行。在 Claude.ai 上，你可以运行它，用户可以下载生成的 `.skill` 文件。

**更新现有技能**：用户可能要求更新现有技能而非创建新技能。在这种情况下：
- **保留原始名称**。记录技能的目录名和 `name` 前置元数据字段——保持不变。例如，如果安装的技能是 `research-helper`，输出 `research-helper.skill`（不是 `research-helper-v2`）。
- **编辑前复制到可写位置**。安装的技能路径可能是只读的。复制到 `/tmp/skill-name/`，在那里编辑，然后从副本打包。
- **如果手动打包，先暂存在 `/tmp/`**，然后复制到输出目录——直接写入可能因权限问题而失败。

## 协作（Cowork）专用说明

如果你在协作环境中，主要注意：

- 你有子 agent，所以主工作流（并行启动测试用例、运行基线、评分等）都能工作。（但如果遇到严重的超时问题，可以串行而非并行运行测试 prompt。）
- 你没有浏览器或显示器，生成评测查看器时使用 `--static <output_path>` 生成独立 HTML 文件而非启动服务器。然后提供一个链接让用户点击打开 HTML。
- 不管出于什么原因，协作环境似乎容易导致 Claude 在运行测试后不生成评测查看器，所以再次强调：无论你是在协作环境还是 Claude Code 中，运行测试后，你应该始终使用 `generate_review.py`（而不是自己编写定制的 HTML 代码）生成评测查看器，让人类在你自己修改技能之前先查看示例。再说一次：在你自己评估输入之前先生成评测查看器！你要尽快让人类看到结果！
- 反馈方式不同：由于没有运行中的服务器，查看器的"提交所有评审"按钮会将 `feedback.json` 作为文件下载。你可以从那里读取它（可能需要先请求访问权限）。
- 打包有效——`package_skill.py` 只需要 Python 和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）在协作环境中应该能正常工作，因为它通过子进程使用 `claude -p` 而非浏览器，但请在完全完成技能制作且用户确认满意后再进行。
- **更新现有技能**：用户可能要求更新现有技能而非创建新技能。请遵循上方 Claude.ai 章节中的更新指南。
