# Session Distiller — 从会话中提炼技能

一个元技能（meta-skill），用于分析当前会话内容，自动发现并提取可复用的技能模式，输出为一个或多个完整的 skill 定义。

---

## 核心流程

```
回顾会话 → 识别模式 → 聚类分组 → 与用户确认 → 生成技能 → 交付
```

---

## 第一步：回顾与分析当前会话

仔细回顾当前会话的全部内容，提取以下信息：

### 1.1 工具使用模式
- 使用了哪些工具？以什么顺序？
- 工具之间的依赖关系和数据流
- 重复出现的工具调用组合

### 1.2 工作流步骤
- 用户请求的完成路径（从输入到输出）
- 关键决策点和分支逻辑
- 用户的修正和迭代（这些往往是最有价值的洞察）

### 1.3 输入/输出格式
- 输入：文件类型、数据格式、用户 prompt 的典型措辞
- 输出：文件类型、格式、结构、质量标准

### 1.4 领域知识
- 会话中涉及的专业知识
- 不是从代码中能直接推导出的隐性知识
- 特定框架/库/API 的使用技巧

### 1.5 用户的修正与反馈
- 用户在哪里纠正了 Claude 的做法？这些纠正揭示了什么隐含需求？
- 哪些做法被认可？哪些被否定？

---

## 第二步：识别候选技能

根据分析结果，识别可以被提炼为独立技能的模式。一个好的候选技能应该：

| 标准 | 说明 |
|------|------|
| **可复用** | 不是一次性任务，未来还会遇到类似场景 |
| **有边界** | 可以清楚地定义"什么时候用"和"什么时候不用" |
| **有增益** | 有了这个技能后，Claude 的表现会明显优于没有技能时 |
| **可独立** | 可以作为独立的技能运行，不强依赖其他技能 |

### 模式分类

会话中常见的可提炼模式类型：

1. **工作流型**：一系列有固定顺序的步骤（如"先读取 → 分析 → 转换 → 输出"）
2. **格式型**：特定的输出格式或模板（如"用这种结构写报告"）
3. **领域型**：特定领域的专业知识和最佳实践
4. **工具组合型**：特定工具的组合使用模式
5. **质量标准型**：特定的质量要求和检查清单

### 拆分 vs 合并

- 如果一个会话涉及多个独立的能力域 → **拆分为多个技能**
- 如果多个步骤紧密耦合、单独使用无意义 → **合并为一个技能**
- 如果某个步骤可以被多个场景复用 → **单独提取**

---

## 第三步：与用户确认

向用户展示你识别到的候选技能列表。格式如下：

```
我从这次会话中识别到了 N 个可提炼的技能：

1. **<技能名称>** — <一句话描述>
   - 来源：<会话中的哪些环节>
   - 触发场景：<什么时候使用>
   - 复用价值：<为什么值得固化>

2. **<技能名称>** — ...

你觉得这些合适吗？要增删或调整吗？
```

等待用户确认后再开始生成。用户可能：
- 合并某些候选技能
- 去掉某些（太简单/太特殊的）
- 补充遗漏的
- 调整优先级（先做哪个）

---

## 第四步：生成技能

对每个确认的技能，生成完整的技能目录。

### 4.1 技能目录结构

```
skill-name/
├── SKILL.md          # 必须
├── references/       # 可选 — 按需加载的参考文档
├── scripts/          # 可选 — 确定性/重复性任务的脚本
└── assets/           # 可选 — 模板、字体等资源文件
```

### 4.2 编写 SKILL.md

#### 前置元数据

```yaml
---
name: <skill-identifier>
description: "<功能描述 + 触发场景描述。写得稍微积极主动一些，对抗欠触发。>"
---
```

description 是最关键的字段——它决定了技能何时被触发。要点：
- 包含功能描述（做什么）和触发描述（什么时候用）
- 列举具体的触发短语和场景
- 稍微"积极主动"——宁可多触发，不可少触发
- 说明什么时候不该使用（如果容易混淆）

#### 正文内容

从会话中提炼以下内容写入正文：

1. **概述**：一段话说明这个技能做什么
2. **工作流/步骤**：按顺序列出操作步骤
3. **关键决策点**：条件分支和判断逻辑
4. **输出格式**：模板或示例
5. **注意事项**：从用户纠正中提炼的"踩坑点"
6. **示例**：用输入/输出格式展示

#### 编写原则

- 正文控制在 **500 行以内**；超出部分移到 `references/` 目录
- 使用**祈使句**
- 解释**为什么**而非仅仅说"必须"
- 从会话的实际操作中提炼，不要凭空发明步骤
- 保留用户修正中的关键洞察（这些是最有价值的部分）

### 4.3 从会话到技能的提炼技巧

| 会话中的原始内容 | 技能中应该怎么写 |
|-----------------|----------------|
| Claude 试了方法 A 失败，换成方法 B 成功 | 直接写方法 B，可以备注"避免方法 A 因为…" |
| 用户纠正："不要用 X，用 Y" | 写成指令："使用 Y（不要用 X，因为…）" |
| 反复调试某个配置 | 提炼出正确配置，附上常见错误 |
| 长时间的探索和搜索 | 提炼出关键路径，列出需要查找的文件/位置 |
| 用户补充的背景知识 | 转化为技能中的领域知识段落 |

---

## 第五步：交付与后续

### 交付

1. 在技能目录中创建所有文件
2. 向用户展示生成的技能摘要
3. 如果生成了多个技能，说明它们之间的关系

### 建议后续步骤

根据技能的复杂度和类型，建议用户：

- **简单技能**："技能已生成，你可以直接试用。如果需要调整，随时告诉我。"
- **复杂技能**："建议用 skill-creator 的评测流程跑几个测试用例，验证效果。"
- **多个技能**："建议按优先级逐个测试和调优。"

如果当前环境中已安装了 skill-creator 或 skill-creator-zh，可以直接建议用户使用它来运行评测。

## 特殊场景

### 会话内容太少

如果当前会话内容不足以提炼出有意义的技能（例如只有一两轮简单问答），坦诚告知用户：

"这次会话的内容比较简单，提炼出的技能可能不比直接用 Claude 好多少。建议：
1. 先继续完成你的任务，积累更多操作模式
2. 或者描述你想要的技能，我来从头创建（可以用 skill-creator）"

### 会话涉及敏感信息

提炼技能时，**自动过滤**：
- API 密钥、密码、token
- 个人身份信息
- 内部系统的具体 URL 或 IP
- 将这些替换为占位符（如 `<YOUR_API_KEY>`）

### 用户要求从特定部分提炼

如果用户指定了"只提炼某段对话"或"只提炼关于 X 的部分"，则只聚焦于指定范围。

### 多语言

- 如果会话是中文的，技能正文用中文
- 如果会话是英文的，技能正文用英文
- 如果会话是混合语言，用用户主要使用的语言，或者直接问用户偏好
